# 10. 運用コスト最適化

低トラフィックの社内向け請求書集計ツールという前提で、運用コストを最小化する
ための方針と実装状況をまとめる。金額は構成判断のための **概算レンジ（要各自確認）**
であり、確定見積もりではない。

---

## コストドライバ

| ドライバ | 種別 | 体感インパクト | 対策 |
|---|---|---|---|
| LLM 分類 (OpenRouter/Anthropic) | 従量 | 唯一の変動費。明細が多い月に増える | 既定オフ + バッチ + 重複排除 + キャッシュ |
| ホスティング計算資源 | 固定 | 常時起動。最大の固定費 | 小型 VPS 1 台に集約 |
| インフラ footprint (RAM) | 固定（間接） | 必要 RAM が増えると上位プラン強制 | 未使用サービスを既定起動から除外 |
| ストレージ/帯域 | 従量（小） | PDF は小さく無視できる規模 | 既定の Supabase Storage で十分 |

結論: **(1) AI を絞る → (2) 単一 VPS → (3) footprint 削減** の順で効く。

---

## 1. AI 分類コスト（実装済み）

### 変更点
- **既定オフ**: `AI_ENABLED=false` が既定。AI を呼ばずルールベースのみで動作 →
  **AI 課金ゼロ**。ルールベースだけでも大半の行は確定する設計
  (`services/classifier.py` の信頼度スコアリング)。
- **オプトイン時もバッチ**: 有効化しても `services/ai_classifier.py` は
  **1 PDF = 1 リクエスト**。低信頼度行を 1 回の呼び出しにまとめる
  (旧実装は「低信頼度行ごとに 1 リクエスト」だった)。
- **重複排除**: 同一 `(工種, 備考, 金額符号)` は 1 項目に集約。請求書は
  同種の明細が多数並ぶためヒット率が高い。
- **プロセス内キャッシュ**: 一度判定した `(model, 工種, 備考, 符号)` は
  プロセス存続中は再問い合わせしない。明細は月次で繰り返すため有効。
- **上限**: `MAX_AI_ROWS=200` で 1 リクエストの暴走を抑制。

### コストモデル（概算）
ある月の PDF に低信頼度行が `N` 行、うちユニークな `(工種,備考,符号)` が
`U` 種類あるとき:

| 実装 | API 呼び出し回数 | 相対コスト |
|---|---|---|
| 旧: 行ごと | `N` | 1.0x |
| 新: バッチ+重複排除 | `1`（未キャッシュが在る月のみ） | おおむね `U/N` 以下 |
| 新: 2 か月目以降（キャッシュ命中） | ほぼ `0` | ほぼ 0 |

請求書のように記述が反復するデータでは **実効 8〜9 割超の削減**が見込める。
さらに `AI_ENABLED=false` 運用なら **完全にゼロ**。

### 将来オプション（未実装・任意）
プロセス再起動でキャッシュは消える。恒久キャッシュが必要なら、既存
`ai_classifications` テーブル（`prompt_input`/`ai_response` を保持）を
`(organization_id, 工種, 備考, 符号)` で参照する DB キャッシュを
Python 側に追加する。`SUPABASE_URL`/`SERVICE_ROLE_KEY` は既に
コンテナに渡っているため追加インフラは不要。トレードオフは 1 クエリ分の
レイテンシと実装工数。低トラフィックではプロセスキャッシュで十分なため
現状は見送り。

## 2. インフラ footprint 削減（実装済み）

`deploy/docker-compose.yml` を **削除ではなく Compose profiles** で既定起動
から外した（可逆・差分最小）。

| サービス | 既定 | 理由 | 起動方法（必要時） |
|---|---|---|---|
| `realtime` | 起動しない (`profiles: ["unused"]`) | アプリは Realtime 未使用 | `--profile unused` |
| `imgproxy` | 起動しない (`profiles: ["unused"]`) | 画像変換不要（PDF のみ） | `--profile unused` |
| `studio` | 起動しない (`profiles: ["admin"]`) | 管理画面は常時不要 + 攻撃面 | `--profile admin` |
| `meta` | 起動しない (`profiles: ["admin"]`) | Studio 専用 | `--profile admin` |

付随変更: `storage` の `imgproxy` 依存と画像変換を無効化、`kong` の
`depends_on` から除外サービスを削除、`AI_ENABLED` を `python-api` に伝播。

既定で常時起動するのは: `preflight`(一瞬) → `db`, `auth`, `rest`,
`storage`, `kong`, `python-api`, `web`（+任意で `caddy`）。
重量級の Elixir(realtime)・imgproxy・Studio(Next) が落ちるぶん必要 RAM が
下がり、**より小さい（安い）インスタンスに収まる**。

> 注: Docker のある環境で `docker compose config` による検証を一度実施すること
> （本リポジトリの編集環境には Docker が無いため未実行）。管理操作時は
> `docker compose --profile admin up -d meta studio` で一時起動 → 終わったら停止。

## 3. ホスティング構成の比較と推奨

### A. 単一 VPS 自己ホスト（推奨：コスト最優先）
`deploy/docker-compose.yml`（footprint 削減後）を 1 台に載せる。

- 目安: 2GB RAM クラスで起動可、4GB あれば余裕。月額 **$6〜24 程度（要確認）**。
- 長所: 固定費が低く予測可能。従量課金の事故が無い。データを自社管理。
- 短所: OS/バックアップ/TLS 更新の運用は自前（Caddy で TLS 自動化済み、
  バックアップは `db-data`/`storage-data` ボリュームの定期スナップショットを
  別途用意）。

### B. マネージド併用（運用を楽にしたい場合）
- DB/Auth/Storage: Supabase マネージド無料枠
- Web: Vercel/同等の無料〜低額枠
- Python API: 小型コンテナ 1 つ（常駐 or スケールゼロ）

- 長所: 運用負荷が小さい。小規模なら無料枠内に収まりうる。
- 短所: 無料枠の上限/スリープ/制限に注意。サービスが分散しコスト把握が複雑。
  本リポジトリの自己ホスト前提（kong ゲートウェイ等）と差異が出る。

### C. 既存 `render.yaml`
複数サービスを個別課金する構成になりやすく、低トラフィック社内ツールでは
A より割高になりがち。使うなら最小インスタンス/スケールゼロを選ぶ。

**推奨**: コスト最優先なら **A（単一 VPS）+ `AI_ENABLED=false`**。
予測可能な最小固定費に、必要時だけ AI をオプトインするのが最も安い。

## 4. 運用チェックリスト（コスト観点）

- [ ] `AI_ENABLED=false` を既定とし、必要な期間だけ true にする
- [ ] AI を使う月は `ai_classifications` の `input_tokens/output_tokens`
      合計（`line_index = -1` のサマリ行）で実コストを監視
- [ ] 既定 `docker compose up -d`（profiles なし）で起動し、`realtime`/
      `imgproxy`/`studio`/`meta` が起動していないことを確認
- [ ] VPS は用途に対し最小サイズから開始し、RAM 逼迫時のみ増強
- [ ] `db-data` / `storage-data` ボリュームのバックアップを別途設定
- [ ] 管理は `--profile admin` で一時起動 → 完了後に停止
